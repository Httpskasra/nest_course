در ادامه هر یک از مفاهیم درخواستی شما را **به‌طور کامل، دقیق و با جزئیات** توضیح می‌دهم. ترتیب را همان‌طور که خواسته‌اید، حفظ می‌کنم.

## ۱. آشنایی با Node.js و اکوسیستم آن (بدون Express)

### Node.js چیست؟
Node.js یک **runtime environment** (محیط اجرا) برای زبان JavaScript است که به شما اجازه می‌دهد کدهای JavaScript را **خارج از مرورگر** اجرا کنید.  
قبلاً JavaScript فقط داخل مرورگرها (Chrome, Firefox, ...) اجرا می‌شد و وظایف سمت کلاینت (مانند کلیک، انیمیشن) را انجام می‌داد. Node.js با استفاده از موتور **V8** (موتور JavaScript کروم) این قابلیت را به سرورها و خط فرمان آورده است.

#### اجرای JavaScript خارج از مرورگر
- می‌توانید با دستور `node filename.js` یک فایل JS را اجرا کنید.
- به اشیایی مثل `window`, `document`, `alert` دسترسی ندارید (چون این‌ها متعلق به مرورگرند).
- به جای آن، اشیایی مثل `global`, `process`, `fs` (فایل سیستم)، `http` و ... در اختیار دارید.

#### معماری رویداد محور (Event-Driven) و غیرهمزمان (Non-blocking I/O)
- **Non-blocking I/O**: وقتی عملیات ورودی/خروجی (خواندن فایل، درخواست شبکه، پرس و جوی دیتابیس) انجام می‌شود، Node.js منتظر تمام شدن آن نمی‌ماند. بلکه عملیات را به سیستم عامل می‌سپارد و بلافاصله به کار بعدی می‌رود. وقتی عملیات به پایان رسید، یک رویداد (event) صدا زده می‌شود.
- **Event-Driven**: یک **حلقه رویداد (Event Loop)** دائماً بررسی می‌کند که آیا رویدادی رخ داده یا callback ای آماده اجراست. به این ترتیب یک رشته (single thread) می‌تواند همزمان هزاران اتصال را مدیریت کند (برخلاف مدل سنتی که برای هر اتصال یک رشته جدید می‌ساخت).
- **مثال**: اگر در Node.js با `fs.readFile` فایلی را بخوانید، کد بعد از آن بلافاصله اجرا می‌شود، بدون اینکه منتظر بماند تا فایل کامل خوانده شود. وقتی خواندن فایل تمام شد، callback شما اجرا می‌گردد.

### مدیریت بسته‌ها با npm یا yarn
**npm** (Node Package Manager) به‌طور خودکار با Node.js نصب می‌شود. **yarn** یک جایگزین سریع‌تر و ایمن‌تر است اما دستورات مشابهی دارد.

#### `npm init` – ساخت فایل package.json
- با دستور `npm init` در ریشه پروژه، یک فایل **package.json** ساخته می‌شود.
- `npm init -y` تمام سوالات را با مقادیر پیش‌فرض تکمیل می‌کند.
- فایل `package.json` شامل اطلاعات پروژه (نام، نسخه، نویسنده)، اسکریپت‌های قابل اجرا (مثل `start`، `test`) و لیست وابستگی‌ها است.

#### نصب پکیج‌ها
- `npm install express` → پکیج express را نصب می‌کند و در `node_modules` ذخیره می‌کند.
- `npm install -g nodemon` → نصب سراسری (برای استفاده در خط فرمان هر پروژه)
- `npm install --save-dev typescript` → نصب به عنوان وابستگی توسعه (`devDependencies`)

#### تفاوت `dependencies` و `devDependencies`
- **dependencies**: کتابخانه‌هایی که برنامه برای **اجرا در محیط production** به آن‌ها نیاز دارد (مثل express، mongoose، dotenv). با `--save` یا بدون flag نصب می‌شوند.
- **devDependencies**: ابزارهایی که فقط در **حین توسعه** نیاز دارید (مثل typescript، nodemon، jest، eslint). با `--save-dev` نصب می‌شوند.
- هنگام `npm install --production` فقط `dependencies` نصب می‌شوند (برای سرور اصلی).

### ماژول‌ها در Node.js
ماژول یعنی جدا کردن کد به فایل‌های مختلف. هر فایل در Node.js یک ماژول محسوب می‌شود.

#### CommonJS: `require` و `module.exports`
- روش سنتی و پیش‌فرض Node.js.
- برای استفاده از یک ماژول دیگر:  
  ```js
  const fs = require('fs');        // ماژول داخلی
  const express = require('express'); // ماژول نصب شده
  const myModule = require('./myModule'); // ماژول محلی
  ```
- برای صادر کردن:  
  ```js
  // در فایل math.js
  function add(a, b) { return a + b; }
  module.exports = { add };
  ```
- یا صادر کردن مستقیم: `module.exports = add;`

#### ES Modules: `import/export` و نحوه فعال‌سازی
- روش مدرن‌تر که با مرورگرها و نسخه‌های جدید Node.js سازگار است.
- برای فعال‌سازی:
  1. در `package.json` مقدار `"type": "module"` را اضافه کنید.
  2. یا از پسوند فایل `.mjs` استفاده کنید.
- مثال:
  ```js
  // utils.mjs
  export function sayHi(name) {
    return `Hello ${name}`;
  }
  
  // app.mjs
  import { sayHi } from './utils.mjs';
  console.log(sayHi('Ali'));
  ```
- **توجه**: در ES Modules، `__dirname` و `__filename` به‌طور مستقیم در دسترس نیستند (باید از `import.meta.url` استفاده کنید).

### فراخوانی توابع غیرهمزمان (Callback, Promise, async/await)
در Node.js بسیاری از عملیات (خواندن فایل، درخواست HTTP، دیتابیس) غیرهمزمان هستند. سه روش اصلی برای مدیریت آن‌ها وجود دارد:

#### Callback
- قدیمی‌ترین روش. تابعی که به عنوان آرگومان به تابع async داده می‌شود و پس از اتمام عملیات صدا زده می‌شود.
- مشکل: **Callback Hell** (هرم نابودی) زمانی که چندین عملیات تودرتو شوند.
- مثال:
  ```js
  const fs = require('fs');
  fs.readFile('file.txt', 'utf8', (err, data) => {
    if (err) console.error(err);
    else console.log(data);
  });
  ```

#### Promise
- شیءای که نشان‌دهنده **تکمیل یا شکست** یک عملیات async در آینده است.
- روش‌های `then`، `catch` و `finally` را دارد.
- تبدیل callback به Promise: بسیاری از کتابخانه‌ها نسخه Promise محور دارند (مثل `fs.promises`).
- مثال:
  ```js
  const fs = require('fs').promises;
  fs.readFile('file.txt', 'utf8')
    .then(data => console.log(data))
    .catch(err => console.error(err));
  ```

#### async/await
- سینتکس شیرین‌تر روی Promiseها. کد شبیه به همزمان (synchronous) می‌شود.
- یک تابع `async` همیشه یک Promise برمی‌گرداند. کلمه `await` فقط داخل تابع async قابل استفاده است.
- مثال:
  ```js
  const fs = require('fs').promises;
  async function readMyFile() {
    try {
      const data = await fs.readFile('file.txt', 'utf8');
      console.log(data);
    } catch (err) {
      console.error(err);
    }
  }
  readMyFile();
  ```
- **نکته آموزشی**: در Node.js مدرن، استفاده از `async/await` به شدت توصیه می‌شود.

## ۲. مفاهیم پایه شبکه و وب (HTTP)

### مدل کلاینت-سرور
شبکه جهانی وب بر اساس **درخواست و پاسخ** کار می‌کند.

- **کلاینت** (Client): دستگاهی که درخواست می‌دهد. معمولاً مرورگر، اما می‌تواند هر برنامه‌ای باشد (Postman، موبایل، تلویزیون هوشمند).
- **سرور** (Server): دستگاهی که منتظر درخواست‌ها است و پاسخ مناسب را برمی‌گرداند. سرورها معمولاً همیشه روشن و دارای آدرس IP ثابت هستند.
- **API** (Application Programming Interface): مجموعه‌ای از قوانین که مشخص می‌کند کلاینت چگونه با سرور ارتباط برقرار کند. در وب، API‌ها معمولاً از طریق HTTP در دسترس هستند.

#### نقش مرورگر، API، سرور
- **مرورگر**: یک کلاینت خاص که درخواست‌های HTTP می‌فرستد، پاسخ HTML/CSS/JS را دریافت کرده و صفحه را رندر می‌کند.
- **API سرور**: برنامه‌ای که روی سرور اجرا می‌شود و درخواست‌ها را پردازش می‌کند. ممکن است JSON برگرداند (APIهای مدرن) یا HTML (وب‌سایت سنتی).
- **سرور سخت‌افزاری**: ماشینی که برنامه API روی آن اجرا می‌شود.

### پروتکل HTTP (HyperText Transfer Protocol)
HTTP یک پروتکل **درخواست-پاسخ** (Request-Response) و **بی‌حالت** (Stateless) است. یعنی سرور اطلاعاتی از درخواست قبلی به خاطر نمی‌سپارد (مگر اینکه از مکانیزم‌هایی مثل session یا token استفاده شود).

#### متدهای رایج HTTP
- **GET**: دریافت اطلاعات (خواندن). بدون تغییر در داده‌های سرور. (ایمن و توانا – idempotent)
- **POST**: ایجاد یک منبع جدید (مثلاً ثبت‌نام، ارسال فرم، ایجاد پست). معمولاً همراه با بدنه.
- **PUT**: جایگزینی کامل یک منبع. ایدیپوتنت.
- **PATCH**: به‌روزرسانی جزئی یک منبع.
- **DELETE**: حذف یک منبع.

#### کدهای وضعیت (Status Codes)
پاسخ سرور شامل یک کد سه رقمی است که نتیجه درخواست را نشان می‌دهد:

- **۲xx (موفقیت)**  
  - `200 OK` – درخواست با موفقیت انجام شد.  
  - `201 Created` – منبع جدید ساخته شد (معمولاً پس از POST).
- **۳xx (هدایت)** – مثل `301` (انتقال دائم)، `304` (محتوای تغییر نکرده – کش).
- **۴xx (خطای کلاینت)**  
  - `400 Bad Request` – درخواست نادرست (مثلاً JSON نامعتبر).  
  - `401 Unauthorized` – احراز هویت نشده (توکن معتبری نیست).  
  - `403 Forbidden` – دسترسی مجاز نیست حتی با احراز هویت.  
  - `404 Not Found` – منبع وجود ندارد.
- **۵xx (خطای سرور)**  
  - `500 Internal Server Error` – خطای عمومی در سرور (مثلاً خطای برنامه).  
  - `503 Service Unavailable` – سرور قادر به پاسخگویی نیست (ترافیک زیاد یا تعمیرات).

#### هدرها (Headers)
اطلاعات اضافی درباره درخواست یا پاسخ که به صورت جفت‌های کلید-مقدار ارسال می‌شوند.

- **هدرهای رایج درخواست (Request Headers)**:
  - `User-Agent`: نوع مرورگر یا برنامه کلاینت.
  - `Accept`: نوع محتوای قابل قبول (مثلاً `application/json`).
  - `Authorization`: توکن احراز هویت (مثل `Bearer ...`).
  - `Cookie`: اطلاعات نشست.
- **هدرهای رایج پاسخ (Response Headers)**:
  - `Content-Type`: نوع محتوای برگشتی (مثل `application/json; charset=utf-8`).
  - `Set-Cookie`: ایجاد کوکی در مرورگر.
  - `Cache-Control`: راهنمایی برای کش کردن.

#### بدنه (Body)
محتوای اصلی درخواست یا پاسخ.

- **در درخواست** (Request Body): فقط در متدهای POST, PUT, PATCH ارسال می‌شود. می‌تواند:
  - JSON: `{"name":"Ali","age":25}`
  - فرم داده (x-www-form-urlencoded): `name=Ali&age=25`
  - FormData (برای آپلود فایل)
  - متن ساده یا XML
- **در پاسخ** (Response Body): هر محتوایی که سرور برگرداند (HTML، JSON، تصویر، فایل).

### آشنایی با URL و پارامترها
URL (Uniform Resource Locator) آدرس منبع است. ساختار:
```
http://example.com:3000/users/123?role=admin#section
```
- `http` – پروتکل
- `example.com` – دامنه (host)
- `:3000` – پورت (اختیاری، پیش‌فرض HTTP:80، HTTPS:443)
- `/users/123` – مسیر (path)
- `?role=admin` – query string
- `#section` – fragment (سمت کلاینت)

#### Query string
- بعد از علامت `?` می‌آید و پارامترهای کلید-مقدار با `&` جدا می‌شوند.
- مثال: `/search?q=express&page=2`
- در سرور به صورت `req.query` قابل دسترسی است (در Express).
- مناسب برای فیلتر کردن، صفحه‌بندی، جستجو.

#### Path parameters (پارامترهای مسیر)
- بخشی از خود مسیر هستند که به عنوان متغیر در نظر گرفته می‌شوند.
- مثال: در مسیر `/users/:userId`، اگر URL `/users/42` بیاید، مقدار `userId` برابر `42` است.
- در Express به صورت `req.params` در دسترس است.
- مناسب برای شناسایی منابع خاص (مثل دریافت کاربر با ID مشخص).

### مفاهیم REST API (ساده)
**REST** (Representational State Transfer) یک سبک معماری برای طراحی API‌های وب است. اصول ساده آن:

- **منبع (Resource)**: هر چیزی که بتوان نام گذاشت و ذخیره کرد (کاربر، محصول، سفارش). هر منبع با یک **URL منحصربه‌فرد** مشخص می‌شود.
- **استفاده از متدهای HTTP برای عملیات استاندارد روی منابع**:
  - `GET /users` → دریافت لیست کاربران
  - `GET /users/5` → دریافت کاربر با ID=5
  - `POST /users` → ایجاد کاربر جدید (داده در بدنه)
  - `PUT /users/5` → جایگزینی کامل کاربر شماره 5
  - `PATCH /users/5` → به‌روزرسانی جزئی کاربر شماره 5
  - `DELETE /users/5` → حذف کاربر شماره 5
- **بدون حالت (Stateless)**: هر درخواست باید همه اطلاعات مورد نیاز (مثل توکن احراز هویت) را به همراه داشته باشد. سرور وضعیت کلاینت را ذخیره نمی‌کند.
- **ارسال داده به صورت JSON** (اغلب) – می‌تواند XML، HTML یا هر فرمت دیگری باشد.

**مثال ساده REST API برای یک فروشگاه:**
| عملیات | متد HTTP | مسیر (Endpoint) |
|--------|----------|----------------|
| دریافت همه محصولات | GET | /products |
| دریافت یک محصول | GET | /products/42 |
| اضافه کردن محصول جدید | POST | /products |
| حذف محصول | DELETE | /products/42 |

> **نکته آموزشی برای بچه‌ها**: REST API را مثل یک کتابخانه در نظر بگیرید. هر کتاب (منبع) یک URL دارد. می‌توانید کتاب‌ها را ببینید (GET)، کتاب جدید اضافه کنید (POST)، کتابی را عوض کنید (PUT) یا قرض دهید (DELETE). این کارها را با متدهای HTTP انجام می‌دهید.
